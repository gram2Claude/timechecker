# COMMIT_CONVENTION — timechecker

Проект ведётся через **workflow_global_plan** (собственный реестр задач + merge-гейт).
Координатор — единственный, кто мержит в `master`.

## Ветки
- **`master`** — защищённая, релизная. Разработчик в неё **не пушит** (branch protection).
  Сюда мержит только координатор через merge-гейт.
- **`oleg`** — рабочая ветка разработчика (имя = логин Windows в нижнем регистре). Вся работа здесь.

## Процесс разработчика
1. Коммить и пушь **ТОЛЬКО** в свою ветку: `git push origin oleg`.
2. Берёшь задачу → напиши координатору: **«беру TIME-XX в работу»** → он ставит In Progress
   (`timechecker task start TIME-XX`).
3. Готов к слиянию → напиши: **«готово к merge, TIME-XX»** (можно несколько ID) →
   координатор запускает merge-гейт (проверка → согласование → merge в `master`).
4. Per-commit строгой конвенции НЕ требуется. Рекомендуется упоминать `TIME-XX` в сообщении
   коммита — улучшает трассируемость (синк считает долю коммитов с валидным ID).

## TASK-ID
- Формат: `TIME-<число>` (readable-идентификатор задачи в собственном реестре timechecker;
  формат не менялся со времён Plane — старые коммиты остаются валидными).
- Авторитетный сигнал «задача сделана» — merge-коммит координатора в `master`, а не твой коммит.

## Чего НЕ делать
- Не пушить в `master` напрямую (заблокировано protection).
- Не работать в чужих ветках.
