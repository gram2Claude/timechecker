# timechecker — спека проекта (intake ЗАКРЫТ)

> Статус: **INTAKE COMPLETE** — переходим к декомпозиции (`plan`). Финал scope зафиксирует план.

## 1. Цель
Инструмент учёта **реального** рабочего времени сотрудников на корпоративном Windows-сервере (RDP):
что делал сотрудник в течение дня, сколько времени ушло на каждую задачу, когда были простои
(>30 мин), плюс набор продуктивностных метрик. Пилот — на самом управленце.

## 2. Принятые решения (intake)
| Решение | Выбор |
|---|---|
| Стек | **Python + uv** (парсинг jsonl/git, отчёты) |
| Репозиторий | **private** |
| Хранилище | **поэтапно: SQLite сейчас → серверная БД позже.** Доступ через **repository-интерфейс** (DAO) — миграция отдельным спринтом, не переписыванием |
| Набор сигналов | **только output** (Claude-транскрипты + git + Plane + канон плана). OS-слежка (ввод/окна) НЕ ведётся |
| Простой | **разрыв задачной активности ≥30 мин** (нет сообщений Claude / коммитов / прогресса), не ОС-idle |
| Дедлайн | не задан → берём расчётную дату из календаря |
| **Milestone пилота** | **агент собирает сигналы D–H и пишет в SQLite** за тестовый день (метрики/отчёты — следующие эпохи) |
| Уведомление сотрудников | **нет** (скрытый мониторинг — решение работодателя; см. риск в §8) |
| dev_branch | **= имя пользователя Windows** (для пилота: `Oleg`) |

## 3. Ключевое преимущество среды
Работа идёт в **Claude Code + VS Code** с обязательным workflow (`workflow_global_plan` →
задачи с `PLANE-ID` и оценками часов; работа в `dev_branch`; память проекта). Поэтому есть
**структурированные задачные логи** — можно привязывать время к конкретной задаче (`PLANE-ID`)
и сравнивать факт с плановой оценкой `estimate_h`. Точнее и **устойчивее к накрутке**, чем нажатия клавиш.

## 4. Источники данных (output-only; на учётке сотрудника)
| # | Источник | Что даёт |
|---|---|---|
| D | **Транскрипты Claude Code** `~/.claude/projects/<proj>/*.jsonl` | таймстемпы сообщений/тул-вызовов, какой проект, объём (токены) → активные сессии |
| E | **Хуки сессий** (SessionStart/End/Stop) | границы рабочих сессий по проекту |
| F | **Git** (коммиты `dev_branch` + `PLANE-ID`) | когда и над какой задачей шла работа, факт поставки |
| G | **Plane** (переходы In Progress / Done) | официальные старт/финиш задачи |
| H | **Канон плана** `00_<slug>_plan.json` | список задач, `estimate_h` → факт vs план |

> Отброшено выбором output-only: idle-ввод ОС, активное окно, Event Log. Следствие — нет «focus-ratio по приложениям».

## 5. Архитектура (поэтапно)
**Пилот (сейчас):** клиент-агент (Python) собирает D–H на учётке сотрудника → **SQLite** через
repository-интерфейс. Дальше — агрегатор метрик и отчётность (CLI/файл, опц. рапорт в Plane).

**Позже (эпоха-миграция):** центральный сервис + серверная БД; агенты шлют агрегаты по сети.
Развилки решим тогда: СУБД (Postgres рек.), хостинг, транспорт/аутентификация (HTTPS+ключ рек.),
веб-дашборд vs файлы. Repository-интерфейс изолирует смену SQLite → серверная БД.

## 6. Метрики
1. **Задачи за день** — список `PLANE-ID` (D+F+G+H).
2. **Время на задачу** — атрибуция активных сессий; факт-часы **vs** `estimate_h`.
3. **Простои ≥30 мин** — разрывы задачной активности; кол-во, длительность, таймстемпы.
4. **Span рабочего дня** — первое/последнее output-событие.
5. **Active vs gap** — покрытие дня активными сессиями vs мёртвые разрывы (замена focus-ratio).
6. **Effort-прокси по Claude** — сообщения/тул-вызовы/токены на задачу (труднее накрутить).
7. **Фрагментация** — переключения между задачами; самый длинный focus-блок.
8. **Adherence** — факт-часы vs `estimate_h`; совпадение с активным спринтом.
9. **Гигиена процесса** — запущены ли скиллы; на своей ли `dev_branch`; доля коммитов с `PLANE-ID`.

## 7. Оговорки
- Атрибуция надёжна при «одна задача/проект на сессию» + теги `PLANE-ID` в коммитах.
- Анти-гейминг: комбинируем Claude + git + Plane, ни одна метрика не в одиночку.
- Простой по output грубее ОС-idle, но ближе к «не работал над задачами».

## 8. Риски и комплаенс (РФ)
- **Скрытый мониторинг без уведомления** (решение принято): повышенный юр. риск по ТК РФ и 152-ФЗ
  (обработка ПДн о трудовой активности). Зафиксировано как осознанный риск работодателя.
- Транскрипты Claude могут содержать рабочий контент/секреты → собирать **метаданные**
  (таймстемпы/проект/объём), а не тело сообщений.
- Чувствительные данные → приватный репо, ограничение доступа, политика ретеншна. При переносе
  в центральную БД — TLS-транспорт и аутентификация агентов.

## 9. Модель данных — ЧЕРНОВИК (детали и финал — на задаче «Схема БД»)
> Это предварительный набросок для понимания; **точные таблицы/поля проектируются и согласуются
> на спец-задаче «Модель данных + SQLite-схема»** (одна из первых задач плана), ДО кода.
> SQLite сейчас, но схема нейтральна к СУБД (готова к переносу на сервер).

Кандидатные таблицы:
- `employee` — `id, windows_username, display_name, dev_branch, active` (учётка ↔ ветка).
- `project` — `id, slug, repo, plane_project_id`.
- `task` — `id, plane_id, plane_identifier, project_id, title, estimate_h, status` (зеркало канона).
- `activity_event` — `id, employee_id, source[claude|git|plane|hook], event_type, project_id, task_id, ts_utc, meta` (сырые нормализованные события, только метаданные).
- `claude_session` — `id, employee_id, project_id, session_id, started_at, ended_at, message_count, tool_calls, tokens`.
- `git_commit` — `id, employee_id, project_id, sha, ts_utc, branch, plane_ids, message`.
- `plane_transition` — `id, task_id, from_state, to_state, ts_utc`.
- `daily_task_time` — `employee_id, work_date, task_id, active_minutes, est_h`.
- `daily_idle` — `employee_id, work_date, gap_start, gap_end, minutes`.
- `daily_summary` — `employee_id, work_date, span_start, span_end, active_minutes, idle_ge30_count, idle_ge30_minutes, tasks_count, hygiene_score`.
- `ingest_run` — `id, employee_id, started_at, finished_at, sources, status` (идемпотентный сбор).

## 10. Когда будет финальная схема БД
Таблицы/поля — это артефакт **задачи «Модель данных + SQLite-схема»** в начале сборки. На ней:
office-hours → спека `01_specs/NN_data_model_spec.md` → план `02_plans/...` с **полным DDL и ER**,
который ты утверждаешь ДО написания кода. Черновик выше — предварительный, для согласования направления.
